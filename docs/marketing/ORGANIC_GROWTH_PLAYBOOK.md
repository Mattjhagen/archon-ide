# Archon Organic Growth Playbook

## North star

Make Archon the browser-based coding agent people trust with real work:

> Choose your model. Give Archon a task. Watch it keep working.

Archon should not market itself as another prompt-to-app toy. The category is a
serious, autonomous development environment that is available from any browser.

## Audience

1. Developers who want agentic coding without changing computers or editors.
2. Founders building prototypes who have outgrown one-shot app generators.
3. Technical teams that need visible reasoning, diffs, terminals, and control.
4. BYOK users who want to choose OpenAI, Anthropic, Gemini, or local models.
5. Mobile and Chromebook developers who cannot install a desktop IDE.

## Messaging hierarchy

### Primary promise

**A coding agent that keeps working.**

### Supporting messages

- A complete AI IDE in the browser.
- Low, Medium, and High reasoning that users can see and control.
- Bring your own model and API key.
- Inspect the plan, progress, terminal, files, and diffs.
- Move between devices without leaving the task behind.

### Claims rule

Only publish a feature claim after it works in production. Label prototypes,
roadmap concepts, and experimental integrations clearly.

## Campaign pillars

### 1. Build in public

Show one meaningful product improvement at a time:

- the broken behavior;
- the exact task given to Archon;
- visible progress while it works;
- the tested result;
- the commit or diff.

Format: 20–45 second screen recording, a short caption, and one direct link.

### 2. The long-task test

Give Archon a task that a one-shot chatbot cannot finish. Examples:

- diagnose and repair a failing responsive layout;
- migrate an API and run the test suite;
- connect authentication and fix redirect edge cases;
- inspect a repository and produce a verified pull request.

The campaign should show elapsed time, steps completed, tests run, and human
interventions. Never imply autonomy that the recording does not demonstrate.

### 3. Model arena

Run the same repository task with OpenAI, Claude, and Gemini. Compare:

- completion quality;
- time;
- tool calls;
- tests passed;
- credits consumed.

Avoid declaring a universal winner. Let the audience choose the model that best
fits the task.

### 4. Browser freedom

Demonstrate the same task continuing across desktop, tablet, Chromebook, and
phone. The visual story is “your development environment is a URL.”

### 5. Honest engineering

Publish technical notes about isolation, credential handling, agent loops,
context management, retries, and cost controls. Trust is the differentiator.

Participate in communities before posting a product link, and follow each
community's self-promotion rules.

## Thirty-day launch sequence

### Week 1 — Establish the problem

- Day 1: Founder story — why one-shot coding chat is not enough.
- Day 2: 30-second “Archon keeps working” product teaser.
- Day 3: Technical post explaining Low, Medium, and High reasoning.
- Day 4: Poll: which provider should handle the first public challenge?
- Day 5: Ship log with a real diff and test result.
- Weekend: Respond to every substantive comment and collect exact objections.

### Week 2 — Prove the product

- Publish the first Long-Task Test.
- Release the uncut YouTube walkthrough.
- Post three short clips: plan, recovery from failure, verified completion.
- Publish the repository, current limitations, and next milestone clearly.
- Invite 20 design partners rather than chasing a large anonymous waitlist.

### Week 3 — Create participation

- Launch “Break Archon”: users submit reproducible coding tasks.
- Select three tasks publicly and report every outcome, including failures.
- Start weekly GitHub Discussion for feature voting.
- Publish the first Model Arena comparison.
- Turn common questions into documentation and search pages.

### Week 4 — Compound

- Publish a transparent monthly scorecard.
- Release a community-selected feature.
- Feature a real user workflow with permission.
- Package the best demo into the permanent homepage hero.
- Announce the next 30-day public challenge.

We kept running into the same problem with AI coding tools: a polished answer

## Demo production template

Every demo must show:

1. The repository and starting failure.
2. The exact user task.
3. Archon's plan and reasoning setting.
4. Files and commands used.
5. At least one moment where verification changes the approach.
6. The final diff.
7. The proof command and result.
8. Credits, elapsed time, and human interventions.

Record a clean 16:9 master at 1440p. Export 9:16 and 1:1 cuts with large captions.
Keep the product UI readable and remove secrets, personal paths, and tokens.

## Growth loops

### Verified task share

After a task completes, generate a share page containing the goal, non-sensitive
steps, diff summary, tests, model, effort, time, and credits. The page links to a
reproducible public repository when available.

### Community challenge

Task submissions become demos; demos become documentation; documentation earns
search traffic; new users submit harder tasks.

### Open-source release loop

Every meaningful release produces:

- GitHub release notes;
- a changelog page;
- one short demo;
- one engineering post;
- one discussion prompt.

## Measurement

Track weekly:

- qualified visits to the product;
- authenticated activations;
- first project opened;
- first real-provider task started;
- verified tasks completed;
- seven-day returning users;
- invite or share rate;
- API cost and gross margin by reasoning level;
- demo view-to-activation conversion;
- organic landing pages receiving impressions.

Initial success is not follower count. It is an increasing number of users who
start a real task, reach a verified result, and return for another task.

## Guardrails

- Never purchase fake followers, engagement, reviews, or backlinks.
- Never spam communities or automate unsolicited direct messages.
- Never use a user's code, task, name, or result in marketing without consent.
- Redact keys, repository secrets, user data, and private paths from recordings.
- Avoid unsupported comparisons such as “better than” or “the best.”
- Reply to criticism with evidence, reproduction steps, and shipped fixes.

## Product dependencies before a broad launch

1. Isolated per-user workspaces and terminals.
2. Durable background tasks with cancel, resume, progress, and budget controls.
3. Real file, search, terminal, test, Git, and diff tool execution.
4. Reliable project import through GitHub or archive upload.
5. Usage metering, rate limits, and abuse controls.
6. A public status page, privacy policy, terms, and security disclosure path.
7. Product analytics with consent and no API-key capture.
8. A complete first-run path that reaches a verified task quickly.
